# 06. ServiceAccount in OpenShift

## What is a ServiceAccount?

A **ServiceAccount** is an identity used by pods and applications inside OpenShift.

Normal users log in to OpenShift by using usernames and passwords, but applications running inside pods use **ServiceAccounts** to access the OpenShift API or to run with specific permissions.

---

## Why do we need ServiceAccount?

ServiceAccount is useful when:

- Application needs special permission.
- Pod needs to access OpenShift API.
- Container image requires a specific user permission.
- Application needs to run with `anyuid` SCC.
- We want to avoid giving extra permission to the default ServiceAccount.

---

## Important Points

- Every project has a `default` ServiceAccount.
- If you do not specify any ServiceAccount, pod uses the `default` ServiceAccount.
- ServiceAccount is created inside a namespace/project.
- ServiceAccount can be associated with SCC.
- SCC controls how a pod can run.
- `anyuid` SCC allows the container to run with any user ID required by the image.

# Security Context Constraints - SCC

## What is SCC?

**Security Context Constraints**, also called **SCC**, control what permissions a pod can use.

SCC controls things like:

- Which user ID the container can run as.
- Whether container can run as root.
- Whether privileged container is allowed.
- What Linux capabilities are allowed.
- Which volumes can be used.
- Whether host network or host path can be used.

---

## Why `anyuid` SCC is required?

Some container images are designed to run with a specific user ID or root user.

By default, OpenShift runs containers with a random high UID for security.

If the image does not support random UID, the pod may fail with permission issues.

In that case, we can use the `anyuid` SCC with a dedicated ServiceAccount.

---

## Check available SCCs

```bash
oc get scc
```

### Check `anyuid` SCC

```bash
oc describe scc anyuid
```

---

## Add `anyuid` SCC to ServiceAccount

### Syntax

```bash
oc adm policy add-scc-to-user <scc_name> system:serviceaccount:<namespace>:<serviceaccount_name>
```

### Command

```bash
oc adm policy add-scc-to-user anyuid system:serviceaccount:alpha:ex280-sa
```

### Short command using `-z`

If you are currently inside the `alpha` project, you can use the `-z` flag.

```bash
oc adm policy add-scc-to-user anyuid -z ex280-sa
```

---

## Verify SCC is added

```bash
oc describe scc anyuid | grep -i ex280-sa
```

or

```bash
oc adm policy who-can use scc anyuid
```

You should observe the ServiceAccount in the output.

If your application is running as `DeploymentConfig`, use the below command.

```bash

```

```bash

```

### Check rollout

```bash
oc rollout status dc/gitlab
```

### Verify

```bash
oc get dc gitlab -o yaml | grep -i service
```

Expected output:

```bash

```

# Method 3: Edit YAML manually

You can also edit deployment YAML manually.

```bash
oc edit deployment gitlab
```

Add or verify the below field under `spec.template.spec`:

```yaml
spec:
  template:
    spec:
      serviceAccountName: ex280-sa
```

Save and exit.

# Question 3:

## Create a new ServiceAccount and assign role permission inside project.

### Requirement

- Create ServiceAccount: `viewer-sa`
- Project: `alpha`
- Assign `view` role to this ServiceAccount.
- Verify permission.

---

## Solution

### Create ServiceAccount

```bash
oc create sa viewer-sa -n alpha
```

### Assign `view` role

```bash
oc adm policy add-role-to-user view system:serviceaccount:alpha:viewer-sa -n alpha
```

or if you are inside project `alpha`:

```bash
oc adm policy add-role-to-user view -z viewer-sa
```

# Troubleshooting

## 1. Pod is not running

Check pod status:

```bash
oc get pods
```

Check logs:

```bash
oc logs pod_name
```

Check events:

```bash
oc events
```

---

## 2. Permission denied error

If application logs show permission errors, check whether the image needs `anyuid` SCC.

```bash
oc logs pod_name
```

Then check ServiceAccount:

```bash
oc get pod pod_name -o yaml | grep -i service
```

---

## 3. Check whether ServiceAccount has anyuid SCC

```bash
oc adm policy who-can use scc anyuid
```

or

```bash
oc describe scc anyuid | grep -i ex280-sa
```

---

## 4. Deployment still using default ServiceAccount

Check deployment YAML:

```bash
oc get deployment gitlab -o yaml | grep -i service
```

If output shows:

```bash
serviceAccountName: default
```

then set the correct ServiceAccount:

```bash
oc set serviceaccount deployment/gitlab ex280-sa
```

---

## 5. New pod is not created after changing ServiceAccount

Restart rollout:

```bash
oc rollout restart deployment/gitlab
```

Check rollout:

```bash
oc rollout status deployment/gitlab
```

# Exam Notes

- ServiceAccount is used by pods.
- User account is used by humans.
- If no ServiceAccount is specified, pod uses `default`.
- SCC is not normally assigned directly to a project.
- SCC can be assigned to a ServiceAccount.
- `anyuid` SCC allows pod to run with any user ID required by the image.
- Use `oc adm policy add-scc-to-user anyuid -z serviceaccount_name`.
- Use `oc set serviceaccount deployment/name serviceaccount_name` to update deployment.
- Always verify with `oc get deployment name -o yaml | grep -i service`.
- After changing ServiceAccount, a new pod should be created.
- Check logs and events if pod is not running.
